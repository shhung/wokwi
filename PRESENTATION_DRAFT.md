# worki 專案開發歷程回顧
*(簡報草稿 / Presentation Draft)*

## Slide 1: 專案概述與目標
**標題：worki 嵌入式系統開發專案**
*   **目標硬體**：STM32 Blue Pill (STM32F103C8)
*   **開發環境**：Wokwi 模擬器
*   **核心功能**：
    *   整合 I2C LCD2004 顯示器
    *   整合 I2C DS1307 RTC 即時鐘
    *   整合 1-Wire DHT22 溫溼度感測器
*   **設計挑戰**：
    *   嚴格的時序要求：1Hz 精確更新 Serial 與 LCD
    *   底層驅動自製 (Bare-metal)：不依賴高階函式庫
    *   處理模擬器環境下的資源競爭與硬體狀態異常

## Slide 2: 架構演進：從 Super Loop 到狀態機
**標題：解決系統掛起與時序競爭**
*   **初期問題**：
    *   採用單一 Super Loop 連續執行所有 I2C 通訊。
    *   導致 I2C 總線瞬間負載過大，模擬器時序崩潰，程式執行 1 分鐘後死當。
*   **解決方案：間隔式狀態機 (Interspaced FSM)**：
    *   將「RTC 讀取 + 4行 LCD 更新」拆分為 5 個獨立狀態。
    *   每次 `loop()` 僅執行一個狀態，並設定 50ms/100ms 的強制間隔。
*   **成效**：
    *   將 600ms 的密集 I2C 傳輸分散，徹底解決總線阻塞。
    *   搭配計時補償邏輯 (`last1sTask += 1000`)，達成極度精準的每秒更新。

## Slide 3: 底層重構：邁向硬體控制器
**標題：從 Bit-banging 到 Hardware I2C**
*   **初期實作 (Bit-banging)**：
    *   為了避開初期對硬體狀態機的不熟悉，採用純軟體模擬 I2C。
    *   雖然穩定，但耗費 CPU 資源。
*   **重構要求 (Hardware I2C1)**：
    *   直接操作 STM32 暫存器 (`CR1`, `DR`, `SR1`, `SR2`)。
    *   設定 PB6/PB7 為 Alternate Function Open-Drain。
*   **設計韌性 (Robustness)**：
    *   實作微秒級的硬體 Flag 逾時保護 (`i2c_wait_flag`)，確保通訊失敗時能安全跳出，不造成系統死鎖。

## Slide 4: 關鍵挑戰一：硬體 I2C 的讀取死鎖
**標題：精確對齊 ACK/STOP 時序**
*   **現象**：
    *   切換至硬體 I2C 後，寫入成功，但讀取多位元組時，除了第一個 Byte 外全為 `0x00`。
*   **根本原因分析 (RM0008 規範)**：
    *   在接收最後一個位元組時，原程式在「讀完資料後」才發送 STOP。
    *   導致 Slave 收到非預期的 ACK，繼續霸佔總線，造成死鎖。
*   **突破性修正**：
    *   重構 `i2c_read(bool ack, bool stop)`。
    *   精準控制在「等待到 RXNE 但**尚未讀取資料暫存器 (DR)**」之前，提前清除 ACK 並觸發 STOP。徹底解決通訊卡死。

## Slide 5: 關鍵挑戰二：隱蔽的 RTC 狀態異常
**標題：無效日期偵測與初始化**
*   **現象**：
    *   通訊恢復後，Serial 依舊顯示 `--:--:--`，判定為錯誤狀態。
*   **根本原因分析**：
    *   透過除錯程式讀出原始資料 `43 0 0 0 0 0 0`。
    *   發現時鐘確實運行中 (CH=0)，但內部日期為全 0 (無效日期)。原先的初始化邏輯僅檢查 CH 位元，導致被略過。
*   **突破性修正**：
    *   實作 **Full Validity Check**：不只看 CH 位元，直接驗證讀取回來的日期是否合法。
    *   若不合法，強制寫入預設時間並重新啟動時鐘，確保系統能自我修復。

## Slide 6: 結論與專案里程碑
**標題：達成 STABLE_HW_I2C_FSM**
*   **極致的穩定性**：
    *   成功結合了硬體 I2C 控制器的高效能與 FSM 狀態機的非阻塞優勢。
*   **精確的輸出合規**：
    *   LCD 與 Serial 輸出完全符合需求規範，具備即時狀態指示 (`OK` / `ERR`)。
*   **AI 協作亮點**：
    *   透過精確的 Prompt 導引，逐步從架構設計、底層除錯到時序最佳化。
    *   利用客製化診斷工具 (`hw_i2c_debug.ino`) 釐清硬體行為，避免盲目修改。
