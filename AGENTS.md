## 核心開發流程：Bottom-Up 穩定架構與逐層驗證
必須嚴格遵循「測試通過一層，才開發下一層」的原則，禁止跨層開發與假設性整合。每一層都必須提供獨立的驗證程式碼。

**1. Tier 1: Bus & HAL Layer (匯流排與硬體抽象層)**
*   **開發要點**：實作底層通訊（如 I2C, SPI, UART, 1-Wire 等）或基礎引腳控制。所有硬體等待迴圈 **必須** 具備基於時間的硬性 Timeout 機制，並盡可能實作總線物理恢復 (Bus Recovery) 邏輯。
*   **驗證方法**：撰寫獨立的通訊壓力測試（如總線掃描器、回環測試 Loopback Test、頻繁的暫存器讀寫）。
*   **通過準則**：在最高預期頻率下，連續執行通訊測試不發生死鎖 (Deadlock)、不掉線，且能正確觸發 Timeout。

**2. Tier 2: Device Driver Layer (裝置驅動層)**
*   **開發要點**：基於 Tier 1 封裝特定感測器、顯示器或致動器的操作 API。驅動層內部 **嚴禁** 使用會影響系統即時性的長阻塞延遲 (Long Blocking Delay)。
*   **驗證方法**：撰寫最小測試單元 (Minimal Test Sketch)，單獨呼叫該驅動讀取設備 ID、寫入測試畫面或獲取單筆原始數據。
*   **通過準則**：設備初始化必定能正確判斷成功或失敗。基本讀寫在單獨測試中 100% 成功，且當設備物理斷線時，驅動能安全回傳錯誤狀態而不崩潰。

**3. Tier 3: Task Scheduler & Logic Layer (任務排程與邏輯層)**
*   **開發要點**：負責多個硬體模組間的資源調度與業務邏輯。必須採用非阻塞式 (Non-blocking) 定時器。對於耗時較長的高負載任務，必須強制拆分為狀態機 (FSM) 分段間隔執行。
*   **驗證方法**：透過除錯介面（如 Serial）高速輸出各任務執行的時間戳記，觀察任務是否準時觸發、耗時是否符合預期。
*   **通過準則**：多個任務並行運作時，無任何任務發生互相霸佔或嚴重阻塞。時間漂移在系統容許的 ±% 範圍內。

**4. Tier 4: Application & System Layer (應用與系統整合層)**
*   **開發要點**：整合所有模組以達成最終產品功能。必須嚴格管控記憶體，全面禁止可能造成記憶體碎片化的動態配置（如過度使用 `String`），優先使用靜態/全域緩衝區。
*   **驗證方法**：進行長時間運行壓力測試 (Burn-in Test)；人為製造極端/邊界條件（如瘋狂觸發 UI、隨機拔插感測器）。
*   **通過準則**：系統連續運行數小時（或更久）無記憶體洩漏、無意外重啟、輸出無截斷/亂碼，且能在遭受干擾後自我恢復。

## 專案特定技術規則 (Technical Best Practices)
- **記憶體管理**：所有格式化緩衝區必須宣告為 `static` 或全域，避免 Stack Overflow。
- **序列埠穩定性**：在輸出長字串或關鍵 Log 後，應呼叫 `Serial.flush()` 以應對模擬器 UI 卡頓。
- **時序補償**：任務排程應使用 `lastTask = now` 以避免在系統凍結後產生「任務追趕死循環」。
- **浮點數處理**：優先考慮「定點數運算」（例如數值 * 10 存為 int），降低運算開銷並避免 `sprintf` 的浮點溢位。
