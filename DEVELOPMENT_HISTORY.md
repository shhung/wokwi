## 2026-05-18: 程式架構與資料結構設計

### 使用者需求 (Prompt)
- 目前已有基本 Serial 輸出，確認 Super loop 運作正常。
- 在接續開發前，要求說明更詳細的程式架構，包含預計如何分層、設計怎樣的資料結構。

### 關鍵決策與架構規劃
**1. 軟體分層設計 (Layering)**
為維持 Bare-metal 的極簡性同時保持可維護性，程式將分為三層：
- **硬體抽象層 (HAL/Bus Layer)**:
    - 負責最底層的時序與暫存器操作。
    - `i2c_start()`, `i2c_write()`, `i2c_read()`
    - `onewire_reset()`, `onewire_write_bit()`, `onewire_read_bit()`
- **裝置驅動層 (Device Driver Layer)**:
    - 利用 Bus Layer 提供的 API 與具體感測器/顯示器溝通。
    - `ds1307_read_time()`
    - `pcf8574_lcd_write_cmd()`, `pcf8574_lcd_print()`
    - `dht22_read_data()`
- **應用邏輯層 (Application Layer)**:
    - `setup()` 與 `loop()` 所在，負責任務排程 (1000ms/5000ms)、狀態更新與畫面刷新邏輯。

**2. 資料結構設計 (Data Structures)**
使用 `struct` 來統一管理設備狀態，避免全域變數散落。
- **RTC 結構**: `struct RtcTime { uint8_t year, month, day, hour, min, sec; bool ok; };`
- **DHT 結構**: `struct DhtData { float temp; float humd; bool ok; };`

- **LCD 佈局調整**：將 LCD 列印邏輯進一步簡化，確保不會因為字串處理耗時過長而影響 Super Loop 的時序。

## 2026-05-18: 將 I2C 重構為 Bit-Banging (軟體模擬)

### 使用者需求 (Prompt)
- 回報 DHT22 已經成功讀取數值，但時間依然無變化且 LCD 沒有畫面。

### 關鍵決策
- **根本原因分析**：硬體 I2C 控制器在模擬器中常常因為時脈配置（如 APB1 時鐘速率與 Arduino Framework 預期不同）或是狀態機的實作瑕疵，導致通訊雖然沒有死當，但資料完全沒有成功傳遞，導致 I2C  設備無回應。
- **重構方案**：為了符合「自行實做 bus driver」的精神，同時徹底繞過所有硬體控制器的不穩定因素，我將整個 I2C 匯流排層重寫為 **Bit-Banging (純軟體模擬)**。
- **實作細節**：
    - 直接操作 `GPIOB_BSRR` 與 `GPIOB_BRR` 控制 PB6 (SCL) 與 PB7 (SDA)。
    - 將腳位設為 General Purpose Open-Drain (CNF=01, MODE=01)。
    - 嚴格控制高低電位的微秒級延遲，完美模擬 I2C 的時序協議。
    - 所有讀寫操作現在都是同步且具備確定性的。

## 2026-05-18: 系統穩定性修復與 LCD 位址偵測

### 使用者需求 (Prompt)
- LCD 依舊無畫面。
- RTC 時間無變化（停留在 00:00:00）。
- Serial 輸出出現亂碼且頻率變慢。

### 關鍵決策
- **RTC 修復**：實作 `ds1307_init`。DS1307 的暫存器 0 有一個 Clock Halt (CH) 位元，若未清除則時鐘不會走動。啟動時檢查並清除該位元。
- **記憶體與亂碼修復**：
    - 將 `loop()` 中的字串緩衝區 `l` 從 21 節點加大至 32 節點，防止 `sprintf` 溢位損毀相鄰的 `DhtData` 記憶體。
    - 簡化 `sprintf` 邏輯，避免複雜的轉義字元。
- **LCD 位址偵測**：在 `setup()` 加入簡易掃描，自動嘗試 `0x27` 與 `0x3F` 兩個常見位址，解決 LCD 無法顯示的問題。
- **I2C 效能優化**：縮短 `i2c_wait_flag` 的超時時間，使系統在設備未回應時能更快跳出，恢復 Super Loop 的執行頻率。

## 2026-05-18: 診斷與修復 I2C 匯流排掛起問題

### 使用者需求 (Prompt)
- 回報系統在初始化後僅輸出一次 LOG，且 LCD 無顯示，代表程式掛起。

### 關鍵決策
- **掛起原因分析**：STM32 Hardware I2C 在模擬器中若遇到未回應的位址（如 LCD 尚未就緒時發送位址），可能會導致 `SR1` 的 Flag 永遠不會被設定，造成 `i2c_wait_flag` 陷入無限死循環。
- **修復方案**：
    1. **Timeout 保護**：在 `i2c_wait_flag` 中加入遞減計數與 `delayMicroseconds`，確保即使硬體未回應，程式也能跳出迴圈。
    2. **初始化延遲**：在 `setup` 與 `i2c_init` 中增加適當延遲，確保外設穩定。
    3. **I2C 軟重置**：在 `i2c_init` 中增加 `SWRST` 控制位元，強制重設 I2C 控制器狀態。
- **LCD 優化**：稍微放寬 `lcd_pulse` 的延遲，確保 LCD 控制器有足夠時間鎖定資料。
